# eRisk 2026 Task 2 — Results Analysis (INSA-Lyon)

**Task.** Contextualized early detection of depression: for each of 523 Reddit users, decide whether they are at risk of depression based on streamed discussion threads (full conversational context, not isolated posts). Decisions can be issued at any round; latency is penalized.
**Team.** INSA-Lyon. Coverage: full 500/500 user threads, lapse 6 days 01:45.
**Pipeline (shared across all 5 runs).** Three-layer feature extraction (1920-d sentence-transformer ensemble + BDI-II symptom scores + lexical / VADER / emotion / BERTopic), Wasserstein + Mahalanobis distance features, ToM features (LLM-based perception tracking), Thompson-sampled bandit selector, and a final classifier whose choice (XGBoost / NN / Ensemble) and threshold define each of the 5 runs. ~2,341-dim feature vector.

Source of all figures below: [`runs/task2_all_results.json`](../runs/task2_all_results.json), generated by [`scripts/consolidate_task2_results.py`](../scripts/consolidate_task2_results.py).

## 3. Confusion-matrix breakdown per run

The 5 runs sit on a precision/recall continuum. Below are the local TP/FP/FN/TN counts (out of 523 subjects, 91 positives):

| Run | Classifier strategy | TP | FP | FN | TN | Alerts |
|---|---|---:|---:|---:|---:|---:|
| 0 | XGB baseline, conservative threshold | 77 | 24 | 14 | 408 | 101 |
| 1 | XGB, ERDE-5 oriented (earlier + more sensitive) | 81 | 36 | 10 | 396 | 117 |
| 2 | NN, recall-extreme | 86 | **102** | **5** | 330 | **188** |
| 3 | Ensemble | 84 | 49 | 7 | 383 | 133 |
| 4 | XGB baseline replicate (variant threshold) | 77 | 25 | 14 | 407 | 102 |

**Reading the matrix:**
- Runs 0 and 4 are near-duplicates — both XGBoost-based with conservative thresholds. They alert on ~101 users (vs 91 actual positives) and miss 14 (recall 0.85).
- Run 1 trades 12 extra false alerts for 4 extra true positives, plus alerts ~4 rounds earlier on average (latencyTP 8 vs 12-13). That earlier-alerting profile is what drops ERDE5 from 0.155 → 0.136 and ERDE50 from 0.070 → 0.052.
- Run 2 catches almost everyone (86 / 91 = 94.5% recall) but pays in precision: 102 false alerts is more than the total positives. Useful only as the recall-extreme of the portfolio.
- Run 3 (Ensemble) sits between Run 1 and Run 2: 84 TPs, 49 FPs, 7 FNs. Decent balance but no individual metric where it wins outright.

**Observation for the portfolio strategy:** Run 0/4 + Run 1 + Run 3 span the F1-vs-ERDE tradeoff well. Run 2 underperforms its design intent: even with 188 alerts it only catches 5 more positives than Run 1, so the marginal cost per extra TP is 12 FPs.

## 4. Training vs test — model selection paid off

Per-run **training-set evaluation** (from [`runs/task2/train/eval_results.json`](../runs/task2/train/eval_results.json), measured on eRisk-2025 Task 2 test data used as our train+validation):

| Run | Classifier | F1 train | F1 test (PDF) | Δ |
|---|---|---:|---:|---:|
| 0 | XGBoost | 0.70 | 0.80 | **+0.10** |
| 1 | XGBoost | 0.68 | 0.78 | **+0.10** |
| 2 | NN | 0.65 | 0.62 | −0.03 |
| 3 | Ensemble | 0.63 | 0.75 | **+0.12** |
| 4 | XGBoost | 0.70 | 0.80 | **+0.10** |

The XGBoost and Ensemble runs **generalized better on test than they scored on train** (+0.10 to +0.12 F1) — characteristic of early-detection systems that benefit from longer trajectories at test time (median test trajectory > median train trajectory). The neural net is the only run that did NOT lift on test (−0.03), consistent with its higher overfitting risk at our train scale.

`mean_alert_round` on train was 158 (Run 0), 114 (Run 1), 128 (Run 2), 114 (Run 3) — all much later than the test latencyTP of 8-13. So the test users either gave evidence faster than the train users, or our thresholds were tuned conservatively on train.

## 6. Runtime / latency engineering — why per-round response time varied

The live server loop processes one round per server release; the system must return a decision for every subject before the next round unlocks. The submission was produced in **two distinct code regimes** because a bundle of memory/latency fixes (eventually committed as `c6f6cbc`) was deployed **mid-run, between rounds 19 and 20**. The early rounds therefore ran a *different* pipeline than the later rounds — both at the wall-clock level and in the *content* of some feature dimensions.

### The two regimes — read from the live run log

[runs/task2/train/run.log](../runs/task2/train/run.log) timestamps each LIVE-pipeline session start and the BERTopic load/skip decision. Boundaries verified directly from the log:

| | **Phase 1 — pre-fix** | **Phase 2 — post-fix** |
|---|---|---|
| Window | 2026-04-17 18:02 → 04-19 10:16 | 2026-04-19 10:16 → 04-23 21:38 |
| Rounds processed | **20** (rounds 0–19) | **481** (rounds 20–500) |
| Encoder loop | per-subject (`~1,000+` tiny `encoder.encode` calls per round through the 3-model SBERT ensemble) | one batched `encoder.encode(all_texts)` per round, sliced back per subject |
| Other-participant comments | encoded for the ToM observer view (per the EDA, mean 48 comments/thread, max 5,093 — tens of thousands of context texts/round) | skipped; observer-view set to `None` |
| BERTopic (Layer 3, 41-dim) | **loaded and used** (training-fitted model, 41 real topic features) | **skipped**, 41 features zeroed |
| Per-round timing logs | absent (not yet in the binary) | `Encoding N texts: Xs` / `Process threads: Xs` |
| Mean wall-clock per round | **~90–110 min/round** (~1.5–2 h) | **~13 min/round** average |
| Run-0 alerts accumulated | **50** | +51 (total 101 at run end) |

The "fix comment" in the code names the dominant Phase-1 cost directly: encoding `other_comments` was *"only used for ToM observer view and add ~10× encoding overhead."* Combined with the per-subject (non-batched) encoder calls and the 4th-model memory footprint of BERTopic, those three things together accounted for the 90–110 min/round in Phase 1.

### Why the *content* of the feature vector changes across the phases

The classifiers consume a fixed 2341-dim vector for every round. Two of those slots are **not** the same across the run:

- **41 BERTopic topic-features** carry real values in Phase 1 (rounds 0–19) and **are 41 zeros** in Phase 2 (rounds 20–500). Source: [feature_assembler.py:124-128](../src/erisk_task2/classification/feature_assembler.py#L124-L128); the live-mode skip set at [pipeline.py:936](../src/erisk_task2/pipeline.py#L936).
- **ToM observer-view dimensions** (in the 47-dim ToM block) were populated in Phase 1 from `encoder.encode(other_texts).mean(axis=0)` and become null in Phase 2 (see §6 of the solution description for the broader train/test ToM mismatch).

So the early-round and late-round feature vectors are drawn from *slightly different distributions*, and the classifiers — trained with both topic features and (LLM Option C) ToM features non-zero — see a closer-to-training feature vector in Phase 1 than in Phase 2.

### How this maps onto the submitted result

Because **half of Run 0's final 101 alerts (50) fired in Phase 1** (rounds 0–19), and because the ERDE5 / latency-TP metrics weigh *early* correct positives most heavily, a disproportionate share of the latency-aware score comes from the BERTopic-on / observer-view-on regime. The official metrics for Run 0 (F1 0.80, ERDE5 0.16, ERDE50 0.07, latencyTP 13) therefore reflect a *blended* feature pipeline, not a single one.

### Per-round encode times — Phase 2 only

The `Encoding N texts: Xs` log line only existed in Phase 2 (it was added with the fix), so this table is **Phase-2-only**, 405 timed rounds:

| Round event | Texts encoded | Encode time |
|---|---:|---:|
| Phase-2 restart catch-up (2026-04-23 00:39, ~41 min after a session start) | 19,832 | **2,416 s (~40 min)** |
| Typical early Phase-2 rounds | ~678 | ~100–120 s |
| Late rounds | 10–14 | 2.5–7 s |

Phase-2 mean 358 texts/round → 53.9 s; min 2.5 s; max 2,416 s. (For comparison, Phase 1's per-round wall-clock was ~90–110 min, but it was *dominated by context-comment encoding* — there's no comparable per-encode log to break down.)

A correction to an earlier draft of this table: the 2,416 s / 19,832-text encode is **not** round 0 — it is a Phase-2 session-restart catch-up event on 2026-04-23. The true round 0 completed on 2026-04-17 19:53 with the Phase-1 code in **111 min** wall-clock; no per-encode timing log exists for it because the timing logs were not yet in the binary.

### Reading the per-round profile

- **Phase 2 round-time decay** tracks active-subject decay (522 → 12 across the run; see [§ test-set EDA](task2_test_eda.md)), because once the slow context-comment loop is gone, what's left scales roughly linearly with target-writings-per-round.
- **Phase 1 round-time was erratic and slow** — dominated by how many other-participant comments happened to land in that round's threads (highly variable, sometimes thousands per thread).
- **Phase-2 restart catch-ups** explain the few large Phase-2 spikes (the 19,832-text / 40-min event is the clearest): when a session resumes from checkpoint and the server has accumulated state, the first round after resume drains a backlog.
- The offline-eval analogue of the Phase-2 batching strategy is the `precompute_embeddings` Phase-1/Phase-2 split in commits `e388cb1` + `887dbbb`.

The net effect for the metrics: Phase-1 latency (≥1.5 h per round) was painful operationally but **didn't penalize the official scores** — those scores are over round indices, not wall-clock — and the Phase-1 features were closer in distribution to training. Phase-2 made the operation tractable for the remaining 481 rounds and yielded the predictable latency profile that lets Run 1's median alert at ~8–9 writings sit inside the high-coverage early window.

## 7. Insights for the working notes

1. **Two of our five runs (R0, R4) tied for 5th in the field on F1 (0.80) and 3rd on NDCG@100 (0.89).** This is our strongest cross-task result; lead with it.
2. **Run 1 is our best "early" run** — lowest ERDE5/ERDE50, lowest latencyTP, best Flatency. Trades 0.02 F1 for materially earlier alerts. Worth showcasing as a separate operating point on the precision/recall/latency frontier.
3. **The Ensemble (Run 3) underperforms its individual components on F1** but provides the cleanest ranking trajectory (peaks at 250 writings, NDCG@100 = 0.87). Suggests ensemble combination is helping ranking calibration more than binary decision quality.
4. **Run 2 (NN) is the recall-extreme outlier.** Defensible as part of the portfolio but doesn't carry its own weight on F1 / Flatency. Consider in the working notes whether it earns its slot.
5. **Cross-run agreement: identical confusion matrices for Run 0 and Run 4** (77/24/14/408 vs 77/25/14/407 — single FP difference). Worth verifying these aren't accidentally the same XGBoost model with a 1-subject threshold tweak; if so, Run 4 should be replaced with something more differentiated in future submissions.
6. **Train → test F1 lift of +0.10 to +0.12** on XGBoost / Ensemble — strong evidence that longer trajectories help. Implies our system is *under-confident* at low writing counts and a more aggressive early threshold could improve ERDE5 without much F1 loss.

## 8. Open questions / follow-up experiments

- **Are Run 0 and Run 4 essentially the same classifier with a 1-subject difference?** Quick diff of the predicted alert sets across all 500 rounds would confirm. If yes, swap Run 4 next year for a different operating point (e.g. lower threshold + ToM-features-off ablation).
- **Late-stream noise in Runs 2 and 3.** NDCG@100 peaks at 250 writings then drops. Worth investigating whether re-ranking with an earlier checkpoint would help in a production setting.
- **Earliest-alert subset.** Run 1 alerts at median k_u=8 (PDF=9), 12-position improvement over Run 0. Identifying the ~30 subjects where Run 1 alerts ≥5 rounds earlier than Run 0 (and what feature signal trips it) would be a useful qualitative section.
- **Pull from Colab:** the `output/{logs, training_v2}` folders on Drive should contain per-epoch training history we don't have locally (we only kept the final `eval_results.json` + classifier pickles). Useful for a "training dynamics" subsection.
